旧v0.2の情報密度を維持しつつ、新しい1-X／2-X／3-X構成へ読み替えた第1章正本候補。

## 1-2　運用主体の定義

| ID | 区分 | 内容 | 主体・責任 | 関連語句 | 備考 |
| --- | --- | --- | --- | --- | --- |
| 1-2 | 運用主体の定義 | 運用者は、ゲーム設計者・GPTユーザーを指す。シーン準備、投稿、投票受付確認、採用行動確定、結果反映依頼、結果投稿までを管理する。プレイヤーは、mixi2の参加者を指し、公開された選択肢に投票してゲーム進行に影響を与える。 | 運用者／プレイヤー | 運用者、プレイヤー | 以後「ユーザー」は原則使わず、「運用者」に統一する。 |

### 1-2補足　責任分離表

| 主体 | 役割 | できること | できないこと |
| --- | --- | --- | --- |
| 運用者 | ゲーム運用主体 | 投稿、判断、確認、採用行動確定、画像生成AIへの投入、結果反映依頼 | 内部状態を直接・無制限に進行更新すること |
| プレイヤー | 投票参加者 | A/B/C/Dへの投票、結果閲覧 | 内部情報参照、採用行動の直接確定、状態更新 |
| GPT | 生成支援レイヤー | 文生成、選択肢案作成、結果文案作成、画像生成プロンプト作成 | 進行更新、画像生成AIへの投入、Actions内部制御 |
| Actions / Backend | 実行・保存レイヤー | 状態保存、状態フラグ管理、結果反映処理 | 投稿文や演出文の最終公開判断 |
| game-state.json | 永続状態の保存先 | 現在状態、選択肢、状態フラグ、内部評価などの保存 | 実行主体ではない |

### 1-2補足　禁止混同

| NG混同 | 理由 |
| --- | --- |
| プレイヤー＝運用者 | 投票する主体と運用判断する主体は分離する。 |
| GPT＝実行主体 | GPTは生成支援レイヤーであり、進行更新主体ではない。 |
| 投稿＝進行更新 | SNS投稿は表示であり、作戦状態を進める処理ではない。 |
| 保存＝進行更新 | 記録と作戦状態の変化は別概念である。 |
| 画像生成＝進行更新 | 画像生成や画像差し替えでは作戦状態を進めない。 |

### 1-3補足　サイクル構造

シーン準備 → 選択肢投稿 → 投票受付 → 採用行動確定 → 結果反映 → 結果投稿 → 次シーン準備

### 1-3補足　基本サイクルの意味

| 観点 | 内容 |
| --- | --- |
| 時間単位 | 1ターンは、シーン準備から次シーン準備までの一連の進行単位である。 |
| 状態単位 | 各工程は状態フラグと対応して管理される。 |
| 投票単位 | プレイヤーは、公開されたA/B/C/Dの選択肢に投票する。 |
| 採用単位 | 投票結果または運用ルールにより、1つの採用行動を確定する。 |
| 進行更新単位 | 作戦状態を進める進行更新は、採用行動確定後の結果反映時にのみ行う。 |
| 詳細参照 | 運用工程の詳細は第2章、状態遷移の詳細は第3章で扱う。 |

### 1-5補足　公開／非公開情報表

| 区分 | 公開情報 | 非公開情報 | 備考 |
| --- | --- | --- | --- |
| 情報種別 | 抽象現在地、状況短文、選択肢文、結果短文、演出 | 実在地名、距離、方角、裏パラメーター名、裏パラメーター数値、票数、内部評価 | 票数は原則非公開。演出上必要な場合のみ「突破派が優勢」など、数値を伏せた傾向表現にする。 |

### 1-5補足　公開制約

| 観点 | 内容 |
| --- | --- |
| 公開原則 | プレイヤーに見せる情報は、抽象化された状況と選択肢に限定する。 |
| 非公開原則 | 内部判断に使う情報、実在地理、数値、票数、内部評価は公開しない。 |
| 画像内テキスト | 投稿文と同じ非公開情報ルールを適用する。 |
| 票数表現 | 数値票数は出さない。必要な場合のみ傾向表現にする。 |
| 目的 | 情報非対称性を維持し、ゲーム性を保護する。 |

| 運用者 | テンプレ画像／キャラ画像を添付し、画像生成AIへ投入し、生成結果を確認する | 非公開情報を画像内に含めない | 第2章 |

### 1-6補足　禁止事項

| NG | 内容 |
| --- | --- |
| GPT画像生成 | 本API運用では、GPTが画像を直接生成しない。 |
| 状態連動画像生成 | 画像生成を理由に作戦状態を進めない。 |
| 内部値描画 | 裏パラメーター、票数、内部評価を画像内に出さない。 |
| テンプレ直接編集前提 | テンプレ画像やキャラ画像を、元画像そのものの編集前提にしない。生成AIで新規生成する運用とする。 |

## 1-7　データ保存・進行更新境界定義

| ID | 区分 | 内容 | 主体・責任 | 関連語句 | 備考 |
| --- | --- | --- | --- | --- | --- |
| 1-7 | データ保存・進行更新境界 | game-state.json に現在地、パラメーター、選択肢、内部評価、状態フラグを永続化する。作戦状態を進める進行更新は applyChoiceResult 実行時の1回のみ行う。選択肢作成やプロンプト作成時に保存する場合でも、進行結果は書き込まない。 | Railway Actions / Backend | game-state.json、applyChoiceResult、保存、進行更新 | 保存と進行更新を分離する。詳細な保存設計は第3章で扱う。 |

### 1-7補足　保存と進行更新の分離

| 区分 | 内容 | 例 | 備考 |
| --- | --- | --- | --- |
| 保存 | メモ、投稿内容、選択肢案、採用行動、状態フラグ、内部情報などを記録する処理 | 投稿内容保存、投票状況メモ、採用行動メモ | 保存のみでは作戦状態は進まない。 |
| 進行更新 | 作戦状態、パラメーター、危険度、ルート進行などを実際に変更する処理 | applyChoiceResult による結果反映 | 採用行動確定後に1回のみ行う。 |

### 1-7補足　不変条件

| 不変条件 | 内容 |
| --- | --- |
| 保存 ≠ 進行更新 | 保存しただけでは作戦状態は変化しない。 |
| 投稿 ≠ 進行更新 | SNS投稿は表示であり、作戦状態を進める処理ではない。 |
| 画像生成 ≠ 進行更新 | 画像生成やプロンプト作成では作戦状態は進まない。 |
| 進行更新の唯一性 | 作戦状態を進める進行更新は、採用行動確定後の applyChoiceResult によってのみ行う。 |

## 1-8　実行制約・エラー／ハング回避定義

| ID | 区分 | 内容 | 主体・責任 | 関連語句 | 備考 |
| --- | --- | --- | --- | --- | --- |
| 1-8 | 実行制約・エラー／ハング回避 | applyChoiceResult は採用行動確定時に1回のみ呼ぶ。GPT Builderプレビューでは画像生成やActions実行のテストを行わない。本番確認は通常チャットまたは専用テスト環境で実施する。選択肢作成、結果文作成、プロンプト生成時は状態フラグを進めない。 | 運用ルール | applyChoiceResult、Actions、エラー回避、ハング回避 | 最重要事故防止ルール。詳細な例外対応は第2章・第3章で扱う。 |

### 1-8補足　実行制約表

| 制約対象 | 内容 | 詳細参照 |
| --- | --- | --- |
| applyChoiceResult | 採用行動確定後に1回のみ実行する。 | 第3章 |
| 選択肢作成 | 選択肢案作成時に進行更新しない。 | 第2章／第3章 |
| 結果文作成 | 結果文作成時に追加の進行更新をしない。 | 第2章／第3章 |
| プロンプト作成 | 画像プロンプト作成時に状態フラグを進めない。 | 第2章／第3章 |
| 画像生成 | 画像生成を理由に進行更新しない。 | 第2章 |
| GPT Builderプレビュー | 画像生成やActions実行のテストを行わない。 | 第2章／第3章 |

### 1-8補足　基本原則

生成 ≠ 状態変化
投稿 ≠ 状態変化
保存 ≠ 状態変化
画像生成 ≠ 進行更新
